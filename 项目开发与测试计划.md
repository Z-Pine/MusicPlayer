## 2. 技术栈确定

| 层级 | 选型 | 说明 |
|------|------|------|
| 框架 | **Tauri v2** (Rust + Web 前端) | 兼顾轻量、开发效率与远期跨平台潜力 |
| 前端 | **Vue 3** + **TypeScript** + **Tailwind CSS** | 组件化、类型安全、样式简洁 |
| 音频播放 | **Web Audio API** + **Howler.js**（或原生 Rust 音频后端） | 视解码需求选择，Tauri 可桥接 Rust 做高性能解码 |
| 数据库 | **SQLite** (通过 `tauri-plugin-sql` 或 Rust `rusqlite`) | 本地数据持久化 |
| 元数据解析 | Rust 侧：`lofty-rs`；前端侧：可桥接 | 支持 ID3 / Vorbis / MP4 等主流标签 |
| 构建与打包 | Tauri CLI | 一键生成 `.msi` 安装包与绿色版 |

> **决策理由**：Tauri v2 在 Windows 下内存占用通常 < 100MB，远优于 Electron；Vue3 + TypeScript 开发体验成熟；Rust 侧可处理 I/O 密集型的扫描与解码任务，与前端 UI 线程天然隔离。

## 3. 模块与架构设计

### 3.1 分层架构

```
┌─────────────────────────────────────────┐
│           前端 UI 层 (Vue 3)              │
│  页面路由 · 组件渲染 · 状态管理 (Pinia)   │
├─────────────────────────────────────────┤
│         前端业务逻辑层 (TS)               │
│  播放控制 · 列表管理 · 歌词解析 · 主题    │
├─────────────────────────────────────────┤
│          Tauri 命令层 (IPC)               │
│   invoke / listen  前后端通信桥接         │
├─────────────────────────────────────────┤
│          Rust 核心服务层                  │
│  音频解码 · 元数据扫描 · 数据库 · 文件 IO │
├─────────────────────────────────────────┤
│          外部依赖层                       │
│  SQLite · 网易云 API · OS 多媒体键        │
└─────────────────────────────────────────┘
```

### 3.2 核心模块划分

| 模块 | 职责 | 主要技术 |
|------|------|----------|
| **MusicScanner** | 递归扫描目录、识别音频格式、提取元数据、生成数据库索引 | Rust: `walkdir` + `lofty-rs` |
| **AudioEngine** | 音频解码、播放控制（播放/暂停/进度/音量）、播放模式切换 | Rust: `rodio` / `symphonia`；或桥接前端 Howler |
| **PlaylistManager** | 播放列表 CRUD、最近播放维护、「我喜欢的」收藏 | Rust + SQLite |
| **LyricsService** | 本地 `.lrc` 加载、网易云 API 歌词搜索、解析与时间同步 | TS: LRC 解析器 + 定时器；Rust: HTTP 客户端 |
| **ConfigManager** | 用户配置、主题、快捷键、音乐库源路径持久化 | SQLite / JSON |
| **GlobalShortcuts** | 注册系统全局热键、多媒体键监听 | Rust: `tauri-plugin-global-shortcut` |
| **UI Shell** | 主窗口布局、导航、播放控制栏、歌词面板 | Vue 3 + Tailwind |

### 3.3 数据库 Schema（核心表）

```sql
-- 音乐库源
CREATE TABLE library_sources (
    id INTEGER PRIMARY KEY,
    path TEXT NOT NULL UNIQUE,
    enabled BOOLEAN DEFAULT 1,
    created_at INTEGER
);

-- 歌曲索引
CREATE TABLE songs (
    id INTEGER PRIMARY KEY,
    path TEXT NOT NULL UNIQUE,
    title TEXT,
    artist TEXT,
    album TEXT,
    album_artist TEXT,
    year INTEGER,
    genre TEXT,
    duration INTEGER, -- 秒
    cover_art BLOB,
    bitrate INTEGER,
    sample_rate INTEGER,
    is_available BOOLEAN DEFAULT 1,
    updated_at INTEGER
);

-- 播放列表
CREATE TABLE playlists (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    created_at INTEGER
);

-- 播放列表歌曲关联
CREATE TABLE playlist_songs (
    playlist_id INTEGER,
    song_id INTEGER,
    position INTEGER,
    PRIMARY KEY (playlist_id, song_id)
);

-- 最近播放
CREATE TABLE recent_plays (
    song_id INTEGER PRIMARY KEY,
    played_at INTEGER
);

-- 用户配置
CREATE TABLE settings (
    key TEXT PRIMARY KEY,
    value TEXT
);
```

### 5.2 各阶段测试重点

| 阶段 | 测试重点 | 测试方法 |
|------|----------|----------|
| Phase 0 | 工程构建稳定性 | CI 自动构建（GitHub Actions），确保每次提交可编译 |

### 5.3 核心测试用例

#### TC-001：音乐库扫描（对应 D-007）
- **前置**：准备测试目录，包含 1000 首混合格式文件（MP3/FLAC/WAV/AAC/OGG），其中 10% 元数据缺失，5% 文件损坏
- **步骤**：添加目录 → 触发扫描 → 观察结果
- **预期**：正确识别 950 首可用文件，损坏文件被跳过并记录日志，元数据缺失文件以文件名为 fallback 标题

#### TC-002：播放状态恢复（对应 D-011）
- **前置**：播放列表中第 10 首歌曲播放到 1:23 时退出应用
- **步骤**：重新启动应用 → 点击「继续播放」
- **预期**：自动加载第 10 首，进度跳转至 1:23，播放模式与音量恢复

#### TC-003：歌词在线匹配（对应 D-021）
- **前置**：本地无歌词，网络通畅
- **步骤**：播放周杰伦《晴天》→ 观察歌词面板
- **预期**：3 秒内从网易云 API 获取并显示歌词，高亮行与歌声同步

#### TC-004：网络降级（对应 D-025）
- **前置**：断开网络连接
- **步骤**：播放任意无本地歌词的歌曲
- **预期**：5 秒后显示「暂无歌词」，UI 不卡顿，不报错崩溃

#### TC-005：全局快捷键（对应 D-026）
- **前置**：应用最小化到任务栏
- **步骤**：按下多媒体键「播放/暂停」→ 按下「下一首」
- **预期**：系统音乐正确暂停/继续/切歌

#### TC-006：性能基准（对应 NF-002 / NF-003）
- **前置**：数据库中已索引 10,000 首歌曲
- **步骤**：冷启动 → 滚动歌曲列表 → 连续播放 2 小时
- **预期**：启动时间 < 2 秒；列表滚动帧率 > 30fps；内存占用 ≤ 150MB；CPU 占用 < 5%

## 7. 风险管理与应对

| 风险 | 影响 | 概率 | 应对策略 |
|------|------|------|----------|
| Tauri v2 API 变更或 Windows 适配问题 | 开发阻塞 | 中 | 锁定 Tauri 版本；关注官方 Migration Guide；关键 Windows API 预留 Qt 降级方案 |
| 网易云 API 封锁或改版 | 歌词功能失效 | 中 | 歌词模块完全解耦；准备 fallback 到本地歌词；评估其他歌词源（如 QQ音乐/酷狗公开接口） |
| 大曲库性能不达标 | 用户体验差 | 中 | 第 6 周即介入性能测试；预留虚拟滚动、数据库索引优化、懒加载策略 |
| 音频解码延迟/格式支持不全 | 播放体验差 | 低 | 优先使用成熟库（`symphonia` / `rodio`）；对疑难格式可调用系统 Media Foundation |
| 全局快捷键与第三方软件冲突 | 功能失效 | 高 | 提供快捷键自定义界面；冲突时友好提示用户 |

## 9. 附录：开发规范

### 9.1 Git 工作流
- 主分支：`main`（始终可编译通过）
- 功能分支：`feature/xxx`
- 修复分支：`fix/xxx`
- 提交信息遵循 Conventional Commits：`feat:` / `fix:` / `perf:` / `test:` / `docs:`

### 9.2 代码规范
- **Rust**：`cargo fmt` + `cargo clippy` 零警告
- **TypeScript**：ESLint + Prettier，严格模式开启
- **命名**：Rust 用 `snake_case`，TS/Vue 用 `camelCase` / `PascalCase`

### 9.3 文档要求
- 每个 Rust `pub fn` 提供文档注释（`///`）
- 复杂业务逻辑在代码中附注释说明决策原因
- 用户可见的变更在 `CHANGELOG.md` 中记录
