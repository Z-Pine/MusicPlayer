# MusicPlayer 项目审查报告

> 审查日期：2026-05-09
> 审查范围：后端 Rust 代码、前端 Vue/TS 代码、需求规格说明书（SRS v1.0）
> 审查目标：识别代码缺陷、架构风险、需求遗漏与前后端不一致问题

---

## 1. 审查结论（TL;DR）

项目已完成 **Tauri + Vue + SQLite 的工程骨架搭建**，后端数据库层和 IPC 接口设计较为完整，但存在以下关键问题：

1. **音频播放引擎完全缺失**——音乐播放器最核心的播放功能尚未实现（前后端均无音频解码/播放能力）
2. **前端为"空壳状态"**——所有界面组件均为占位实现，未调用任何后端 API
3. **前后端数据模型类型不一致**——`Song.id` 前端为 `string`，后端为 `i64`
4. **需求覆盖率不足**——P0 级需求中，全局快捷键、继续播放、歌词显示、主题切换均未落地
5. **后端扫描存在性能隐患**——逐文件重复获取数据库锁，大曲库扫描将极慢

## 5. 关键风险与建议

### 5.1 🔴 最高优先级（阻塞发布）

| 问题 | 影响 | 建议措施 |
|------|------|----------|
| **音频播放引擎缺失** | 播放器无法播放音乐 | 方案 A：前端用 HTML5 `<audio>` + Howler.js（开发快，但功能受限）<br>方案 B：Rust 端用 `rodio`/`symphonia` 做解码，通过 Tauri 事件向前端推送音频数据（性能好，开发复杂）<br>**推荐方案 A 做 MVP，方案 B 做远期优化** |
| **前端未调用后端 API** | 所有后端 Command 成「死代码」| 为每个核心功能补全前端 `invoke` 调用：扫描、获取歌曲、获取播放列表、播放状态读写 |
| **前后端 Song ID 类型不一致** | 数据流转时类型混乱 | 统一使用 `number`（前端）对应 `i64`（后端），Tauri IPC 会自动处理 JSON Number |

### 5.2 🟡 高优先级（影响体验）

| 问题 | 建议措施 |
|------|----------|
| 扫描性能（逐文件加锁）| 改为"收集所有元数据 → 批量事务写入"模式，或至少缓存 `MutexGuard` |
| 播放模式缺少单曲循环 | 补全 `playMode` 枚举为 4 种：`sequence`/`loop_list`/`loop_single`/`shuffle` |
| 全局快捷键未注册 | 在 `lib.rs` 中配置 `tauri-plugin-global-shortcut`，注册播放/暂停/切歌/音量键 |
| 歌词功能全链路缺失 | 前端增加 Lyrics 组件，先实现本地 `.lrc` 解析，再接入网易云 API |
| 主题系统缺失 | 引入 CSS 变量体系，支持 light/dark 切换，可结合 `prefers-color-scheme` |

### 5.3 🟢 中优先级（长期优化）

| 问题 | 建议措施 |
|------|----------|
| Tailwind CSS 未引入 | 若计划长期维护，建议引入 Tailwind 统一管理主题色和响应式 |
| 封面图 BLOB 传输性能 | 增加缩略图生成，或单独提供封面 Command（`get_cover(song_id)`）|
| 「我喜欢的」收藏 | 增加 `favorites` 表，在 Song 列表行增加星标按钮 |
| 文件夹视图 | 根据 `songs.path` 字段做路径分组，前端用树形组件展示 |
| 任务栏媒体控制 | 使用 Windows Media Control API 或 Tauri 的 OS 集成能力 |
